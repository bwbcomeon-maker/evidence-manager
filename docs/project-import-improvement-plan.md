# 项目批量导入功能调整规划

基于《project-create-vs-import-analysis.md》中梳理的问题，对批量导入功能做全面规划。**本文档仅规划，不涉及代码修改。**

---

## 一、问题与目标

### 1.1 当前存在的问题（回顾）

| 序号 | 问题 | 影响 |
|------|------|------|
| P1 | **has_procurement 不一致**：导入新增项目未设置该字段，库内为 false；单项目创建默认 true。 | 导入项目的 S1 阶段「项目前期产品比测报告」不参与必填，与单项目创建行为不一致，易造成业务理解偏差。 |
| P2 | **导入新增项目无成员**：不写入 auth_project_acl。 | 导入后项目对普通用户不可见，必须由 PMO/管理员在「成员管理」或「批量分配项目」中分配，与单项目“创建即可见”体验不一致。 |
| P3 | **模板无「是否含采购」列**：Excel 仅 3 列（项目令号、项目名称、项目描述）。 | 无法通过导入指定或批量维护 has_procurement，也无法与单项目表单字段对齐。 |
| P4 | **导入允许项目名称为空**：name 空时仅转为 ""，不报错。 | 与单项目「项目名称不能为空」的校验不一致，可能产生无名称项目。 |
| P5 | **导入更新不更新 has_procurement**：update 分支只改 name、description。 | 无法通过重新导入 Excel 批量修正「是否含采购」，与单项目「可编辑基本信息」能力不对齐。 |

### 1.2 调整目标

- **与单项目创建对齐**：导入在「可配置范围内」与单项目创建/编辑的字段与规则一致，避免同一种业务两种行为。
- **可预期、可运维**：模板与说明清晰，校验规则明确，错误信息可定位到行与列。
- **兼容现有使用方式**：在明确变更的前提下，尽量兼容已有 Excel 文件（如仅增加可选列、默认值策略）。
- **保持批量导入的定位**：不改变「仅 SYSTEM_ADMIN/PMO 可导入」「按 code upsert」「不自动加成员」等既有设计，仅补齐字段与校验。

---

## 二、分项规划

### 2.1 has_procurement（是否含采购）

**目标**：导入新增/更新时与单项目创建/编辑在「是否含采购」上可对齐、可配置。

**方案选择**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A | 模板增加第 4 列「是否含采购」，支持 是/否/1/0/Y/N 等，导入时解析并写入。 | 与单项目一致，可批量维护，更新时也可改。 | 模板变更，需更新说明与示例。 |
| B | 不增加列，导入新增时固定为 true（与单项目默认一致），更新时不改。 | 无模板变更，改动小。 | 无法通过 Excel 区分是否含采购，更新时仍无法修正。 |
| C | 不增加列，导入新增时固定为 true；更新时增加「可选列」：若提供则更新 has_procurement，未提供则不改。 | 兼容老模板，新模板可扩展。 | 实现略复杂，需约定列顺序或列名。 |

**推荐**：**A**。  
- 与单项目表单字段一致，业务含义清晰。  
- 更新时也可通过重新导入修正「是否含采购」，便于批量维护。  
- 模板与文档一次性说清楚即可。

**实施要点（规划层面）**：

- 模板：第 4 列表头「是否含采购」，示例行可写「是」或「否」。  
- 解析规则：约定 是/1/Y/true 等视为 true，否/0/N/false/空 等视为 false；空或未识别时建议**默认 true**（与单项目默认一致），并在说明中写明。  
- 新增：`project.setHasProcurement(解析结果)`，不再依赖 DB 默认。  
- 更新：若模板提供该列，则 `existing.setHasProcurement(解析结果)`；若未提供（如旧版 3 列模板），则不改 has_procurement，保持兼容。

---

### 2.2 项目名称 name 校验

**目标**：与单项目「项目名称不能为空」一致，避免产生空名称项目。

**方案**：

- 导入时对 **name** 做非空校验：trim 后为空则视为不合法。  
- 行为：该行记入 errors（如「第 N 行：项目名称为空」），不插入、不更新，inserted/updated 不计数，与「项目令号为空」一致。  
- 可选：对 description 维持现状（允许空），与单项目「描述可选」一致。

**实施要点**：

- 在 `importProjectsFromExcel` 循环内，在 code 校验之后、查库之前，增加对 name 的校验。  
- 错误码/文案与现有 RowError 风格统一，便于前端展示。

---

### 2.3 导入更新时支持 has_procurement（与 2.1 联动）

**目标**：通过重新导入 Excel，可批量修正已有项目的「是否含采购」。

**方案**：

- 与 2.1 一致：若模板包含「是否含采购」列，则 update 分支中除 name、description 外，增加对 `hasProcurement` 的更新。  
- 若模板不包含该列（兼容旧版 3 列）：update 时不修改 has_procurement。  
- 可选：在文档中说明「若需批量修改是否含采购，请使用带第 4 列的新版模板」。

---

### 2.4 导入新增项目是否自动加操作人为成员（auth_project_acl）

**目标**：在「与单项目体验对齐」和「批量导入不强制改变权限」之间做明确选择并文档化。

**方案选择**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| D1 | 导入新增时，为每个新项目插入一条 owner，sys_user_id = 操作人。 | 与单项目一致，导入者立即可见、可管理。 | 大批量导入时，操作人成为大量项目负责人，可能不符合「仅做数据导入、不参与管理」的场景。 |
| D2 | 保持现状：导入不写入 ACL，成员需后续分配。 | 权限清晰，适合「导入后由 PMO 统一分配」的流程。 | 与单项目「创建即可见」不一致，需在说明中强调。 |
| D3 | 可配置：系统或导入参数控制「是否将操作人加为 owner」，默认 off。 | 灵活，可兼顾两种场景。 | 实现与产品逻辑更复杂，需新增配置或参数。 |

**推荐**：**D2（保持现状）**，并辅以**说明与引导**。  
- 批量导入的典型场景是「批量建项目 + 后续统一分配」，与单项目「个人建个人管」不同。  
- 若改为 D1，大批量导入会使操作人成为大量项目 owner，需评估权限与审计需求。  
- 在「批量导入」说明中明确写清：导入的新项目无成员，需在「成员管理」或「批量分配项目」中分配后，相应人员才可见；与单项目创建差异属于设计选择而非缺陷。

**若未来希望对齐单项目**：再考虑 D1 或 D3，并单独做权限与审计评估。

---

### 2.5 模板与接口约定

**目标**：模板列与单项目表单/编辑字段对齐，且兼容旧版 3 列。

**建议**：

- **新版模板列顺序**：  
  1. 项目令号（必填）  
  2. 项目名称（必填）  
  3. 项目描述（选填）  
  4. 是否含采购（选填，是/否等，空则默认是）

- **兼容策略**：  
  - 若 Excel 表头或列数不足 4 列：仍按 3 列解析，has_procurement 行为按「未提供」处理（新增用默认值 true，更新不更新该字段）。  
  - 若第 4 列存在：按 2.1 规则解析并写入/更新。

- **模板下载**：  
  - `GET /api/projects/import/template` 返回的 xlsx 建议提供 4 列表头 + 示例行，并在响应或前端说明中注明「第 4 列可选，不填则默认含采购」。

- **文档**：  
  - 在帮助/产品说明中更新「项目导入」说明：列含义、必填/选填、是否含采购的取值与默认值、错误码与处理建议。

---

## 三、实施阶段建议（仅规划）

### 阶段 1：字段与校验对齐（优先）

- 1.1 模板增加第 4 列「是否含采购」，解析规则与默认值（空则 true）确定并实现。  
- 1.2 导入新增时显式设置 `hasProcurement`（来自解析或默认 true）。  
- 1.3 导入时对 name 做非空校验，空则记 RowError，不插入/不更新。  
- 1.4 导入更新时，若存在第 4 列则更新 has_procurement；否则不更新（兼容 3 列）。

**交付**：导入与单项目在 code/name/description/hasProcurement 上行为一致；旧版 3 列 Excel 仍可用，新项目默认含采购。

### 阶段 2：模板与文档

- 2.1 更新 `downloadImportTemplate` 生成 4 列模板，示例行含「是否含采购」示例值。  
- 2.2 前端导入说明/提示文案更新：列含义、必填项、第 4 列可选及默认值。  
- 2.3 若有独立「项目导入」帮助页，同步更新并注明与单项目创建的差异（如成员需后续分配）。

### 阶段 3：可选增强（按需）

- 3.1 导入结果中可选返回「本批新增项目 id 列表」，便于后续批量分配或脚本对接。  
- 3.2 若产品选择「导入时可将操作人加为 owner」（D1/D3），再单独设计配置项或参数及权限说明。

---

## 四、兼容性与风险

### 4.1 兼容性

- **旧版 3 列 Excel**：仍可导入；新增项目 has_procurement 由默认值（true）决定；更新项目不修改 has_procurement。  
- **新版 4 列 Excel**：第 4 列参与解析与写入；更新时可修改 has_procurement。  
- **API**：`POST /api/projects/import` 入参仍为 multipart file，无需改接口签名；仅服务端解析逻辑与写入字段扩展。

### 4.2 风险与注意点

- **name 必填**：历史上若有「项目名称为空」的 Excel，导入后该行会从「静默写入空字符串」变为「报错不写入」，需在发布说明中注明。  
- **has_procurement 默认值**：若将「空视为 true」与部分业务习惯不符，可在文档中明确，必要时再增加配置项。  
- **大批量导入**：若未来支持「导入时加操作人为 owner」，需评估 500 行批量产生的 ACL 数据量与权限影响。

---

## 五、验收预期（规划层面）

完成阶段 1～2 后，预期达到：

1. 导入新增项目：code、name、description、has_procurement、status、created_by_user_id 与单项目创建逻辑对齐（has_procurement 可由 Excel 或默认值确定）。  
2. 导入更新项目：可更新 name、description，并在提供第 4 列时更新 has_procurement。  
3. 项目名称空时导入报错并记入 errors，不写入库。  
4. 旧版 3 列模板仍可使用，新项目默认含采购；新版 4 列模板可完整维护是否含采购。  
5. 文档与界面说明已更新，明确成员需后续分配等与单项目创建的差异。

---

## 六、总结

| 问题 | 建议方向 | 说明 |
|------|----------|------|
| P1/P3/P5 has_procurement | 模板增列 + 解析 + 新增/更新都支持 | 与单项目对齐，可批量维护，兼容 3 列 |
| P4 name 为空 | 导入校验 name 必填，空则 RowError | 与单项目「项目名称不能为空」一致 |
| P2 无成员 | 保持不自动加成员，文档说明差异 | 符合当前批量导入定位，避免权限膨胀 |

按上述阶段推进即可在**不修改单项目创建逻辑**的前提下，使批量导入在字段与校验上与单项目创建/编辑对齐，并保留兼容性与可维护性。具体实现时再按本规划拆分为具体开发任务与用例即可。
